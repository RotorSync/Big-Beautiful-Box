# Bug Hunt Notes - 2026-05-28

Read-only bug hunt from Codex plus subagents. No fixes were made during the hunt.

## Highest Priority

### P1 - IO-Link recovery/socket calls can stall the safety loop

`dashboard.py` can hold `iol_io_lock` through IO-Link power-cycle sleeps/status polling while the flow-control loop needs that same lock for `read_flow_meter()`. The IOL socket wrappers also do not consistently set timeouts around connect/recv. If IO-Link stalls or recovery starts during flow, auto-shutoff decisions can pause for seconds.

Relevant areas:
- `dashboard.py`: `iol_power_cycle()`, `read_flow_meter()`, `flow_control_loop()`
- `iolhat.py` and `src/iolhat.py`: socket connect/recv calls
- `config.py`: `FLOW_CONTROL_INTERVAL`

Recommended fix:
- Add socket timeouts to IO-Link calls.
- Avoid holding `iol_io_lock` across recovery sleeps.
- Make the safety loop fail closed when data is stale or recovery is active.

### P1 - Stale flow data can still drive auto-stop

Duplicate raw reads can refresh `last_successful_read_time`, while stale detection sets `connection_error`. `_flow_control_process_sample()` currently gates mainly on elapsed timeout and does not directly reject `connection_error`, so cached stale values can still participate in relay decisions.

Relevant areas:
- `dashboard.py`: `read_flow_meter()`, `_flow_control_process_sample()`

Recommended fix:
- Treat `connection_error` / stale duplicate state as disconnected for shutoff decisions.
- Only let fresh or explicitly valid cached samples drive auto-stop.

### P1 - Overlapping relay pulses can shorten pump-stop hold

Auto-stop, serial `PS`, and socket `PS` can all call `pump_stop_relay()`. The function sets GPIO HIGH, sleeps, then sets LOW with no relay lock or owner/expiry guard. If two pulses overlap, the first finisher can set LOW while the second still expects the relay held active.

Relevant areas:
- `dashboard.py`: `pump_stop_relay()`, `start_pump_stop_thread()`, serial `PS`, socket `PS`

Recommended fix:
- Serialize relay control with a lock or single relay worker.
- Track desired relay hold-until time so repeated stop requests extend the hold rather than fight each other.
- Use best-effort LOW in a safe cleanup path.

### P1/P2 - `PS` is not pump stop in all UI modes

Serial `PS` triggers pump stop only in normal dashboard mode. In dialogs, menus, tests, and calibration screens it may cancel, exit, mark a test, or be ignored. If `PS` is meant to be emergency pump stop, the current behavior is inconsistent.

Relevant areas:
- `dashboard.py`: serial routing branches for confirmations, calibration, log viewer, fill history, self-test, full-test, menu, normal mode
- `README.md`: field behavior description
- `web-sim/server.py` and native sim routing

Recommended fix:
- Decide whether `PS` is globally emergency stop or contextual cancel.
- If emergency stop, route `PS` to relay first in every mode, with a separate cancel/back input if needed.

## High/Medium Priority

### P2 - BatchMix can use a stale target when already in mix mode

When a BatchMix payload arrives while already in mix mode, the socket handler can update `mix_requested_gallons` and draw the new number without assigning live `requested_gallons`. Auto-stop still uses `requested_gallons`, so displayed recipe target and shutoff target can diverge.

Relevant areas:
- `dashboard.py`: BatchMix socket handler, `switch_mode()`, flow-control target reads

Recommended fix:
- When accepting BatchMix water needed, update the live target atomically with the preset.
- Add a small test or log assertion for already-in-mix BatchMix updates.

### P2 - Red/off-target thumbs-up can train the learned curve

The UI allows thumbs-up outside the green +/-2 gallon threshold, keeps the screen red, and still records the pending fill. Learning accepts Auto samples up to the broader 5 gallon guard, so an operator-confirmed but visibly bad fill can move the learned curve.

Relevant areas:
- `dashboard.py`: `handle_thumbs_up_press()`, `change_colors_to_green()`, `record_pending_fill()`
- `src/flow_curve.py`: sample error guard

Recommended fix:
- Record the fill/history if desired, but skip curve learning unless the fill is within the green acceptance threshold.
- Log skipped learning reason clearly.

### P2 - Auto-shutoff is latched before relay success is known

The control loop marks `last_alert_triggered` and `auto_shutoff_latched` before relay activation succeeds. If GPIO is unavailable or a relay exception occurs, the fill can still be classified as `Auto`, which can affect history and learning.

Relevant areas:
- `dashboard.py`: `_flow_control_process_sample()`, `pump_stop_relay()`, fill completion classification

Recommended fix:
- Track relay attempt versus relay confirmed active.
- Classify learning samples as Auto only if relay activation succeeded.

### P2 - Self-test manipulates Tk widgets from a background thread

`run_self_test()` starts a background thread that directly calls `results_text.insert()` and `update()`. Tkinter is not thread-safe, so this can cause intermittent hangs or crashes during diagnostics.

Relevant areas:
- `dashboard.py`: `run_self_test()`

Recommended fix:
- Run hardware checks in the worker thread, but send UI updates through `root.after()`.

### P2/P3 - Fresh install can miss required RotorSync module

`rotorsync_bumble.py` imports `src.bluetooth_adapter_selection`, but `install.sh` only copies selected `src` files into `/opt/src`. Existing updated boxes may work because the dashboard updater copies all `src`, hiding the fresh-install failure.

Relevant areas:
- `rotorsync_bumble.py`
- `install.sh`
- `dashboard.py` updater

Recommended fix:
- Make installer copy the whole `src` package or explicitly include every imported module.

## Additional Findings

### P2/P3 - Socket commands are not message-framed

Dashboard/socket helpers use a single `recv(4096)` and parse immediately. TCP can split or coalesce messages, and larger BatchMix/config payloads can be truncated.

Relevant areas:
- `dashboard.py`: socket listener
- `src/socket_handler.py`
- `rotorsync_bumble.py`

Recommended fix:
- Use newline-delimited framing or length-prefixed messages.
- Read until full command terminator before parsing.

### P3 - Calibration can leave normal fill target changed

Tank calibration writes live `requested_gallons` / `fill_requested_gallons` for each calibration step and does not restore the prior fill preset on completion or abort.

Relevant areas:
- `dashboard.py`: calibration workflow

Recommended fix:
- Save previous target/preset when entering calibration.
- Restore it on completion/cancel unless the user explicitly chooses otherwise.

### P3 - Accepted learned curves are dashboard-only in some shared paths

The dashboard uses `active_flow_curve`, but shared calculation helpers default to factory if no curve is passed. Simulator/shared-code validation can disagree with the real dashboard after accepting a learned curve.

Relevant areas:
- `dashboard.py`
- `src/calculations.py`
- `src/flow_curve.py`
- `web-sim/server.py`

Recommended fix:
- Centralize active curve loading for all runtime/sim paths, or explicitly pass the active curve everywhere threshold calculation matters.

### P3 - Native simulator routing differs from field serial routing

Native simulator command routing handles confirmations, menu, and normal mode, but not all real serial branches such as calibration, log viewer, fill history, self-test, full-test, and update mode.

Relevant areas:
- `dashboard.py`: `_sim_send_command()` and serial listener routing

Recommended fix:
- Share a routing function between serial and simulator where possible.
- At minimum, add sim branches for field-critical screens.

### P3 - Calibration persistence is not simulator-safe

Some calibration existence checks use hard-coded `/home/pi` paths while writes are redirected in simulator mode. This can overwrite sim data or duplicate CSV headers.

Relevant areas:
- `dashboard.py`: calibration persistence helpers

Recommended fix:
- Use one path abstraction for exists/read/write in simulator and production.

### P3 - Browser workbench 3-second coast readout is off by 2x

The web sim reports `flow_gpm * 3 / 120`; gallons over 3 seconds at GPM should be `flow_gpm * 3 / 60`.

Relevant areas:
- `web-sim/server.py`

Recommended fix:
- Correct the display math. This is a simulator/control-panel mismatch, not field shutoff logic.

### P3 - Menu color/style metadata has drifted

`MENU_ITEMS` has 14 entries, while the menu palette/comment list has 12 entries and wraps. This is mostly visual/maintenance drift.

Relevant areas:
- `dashboard.py`: `MENU_ITEMS`, `style_menu_item()`

Recommended fix:
- Give every menu item an explicit style or derive colors without stale comments.

## Repo Hygiene

The root contains checked-in backup artifacts such as `dashboard.py.bak`, dated dashboard backups, and `config.py.bak`. This makes review noisy and increases the chance of editing or deploying the wrong copy.

Recommended cleanup:
- Move old backup artifacts out of the repo or delete them in a deliberate cleanup commit.
- Add ignore rules for future local backup files.
